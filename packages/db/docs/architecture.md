# Database Package Architecture

`@complyflow/db` owns Prisma and database mapping.

The database stores persistent Google-backed users, organizations, and organization memberships. Membership roles are `owner` and `member`; both can edit workspace data in the current product surface. Company attributes are flattened onto the `organizations` table, infrastructure/data handling/access sections use one-to-one profile tables, organization data categories live in `organization_data_types`, and provider inventory is relational in `organization_providers` so providers can be added, edited, and removed independently. Controlled vocabulary definitions loaded from Airtable live in `system_code_sets` and `system_codes`; non-system sets are cloned into `organization_code_sets` and `organization_codes` for organization editing. Countries live in the separate `countries` table and use ISO alpha-2 codes. Infrastructure system selections use `organization_providers.provider_id` and `organization_providers.system_type` instead of provider-name columns on `infrastructure_profiles`. `vendor_data_types` links each provider to the organization data categories it processes. Editable template copies live in `templates` and retain their source system template slug for traceability. Generated markdown documents live in `documents`, keep a source hash for stale-state detection, and store private PDF export object paths for authenticated downloads. API-facing payloads remain nested and are validated at API boundaries with shared Zod schemas.

# Database Package Architecture

`@plyco/db` owns Prisma and database mapping. Prisma CLI settings live in `prisma.config.ts`, and the generated Prisma 7 client is emitted to `src/generated/prisma` by the package build/typecheck scripts.

Public beta interest is stored in `waitlist_entries`, independent of users and organizations. Normalized email is unique, and repeat submissions retain the latest optional blocker for that address.

Vocabulary tables persist Airtable hint-display flags and optional code descriptions. Newly cloned or backfilled organization codes inherit system descriptions, while later catalog loads preserve organization-edited descriptions.

Organization-specific advisor rule suppressions live in `organization_rule_suppressions`, keyed uniquely by organization and stable static rule ID. They cascade with the organization; rule definitions and evaluation results remain outside the database.

Explicit infrastructure, AI, and newsletter answers that no provider is used are stored in `infrastructure_profiles.explicit_no_provider_system_types`. They are mapped to `providerId: "none"` in the relevant profile payload and are not organization-provider inventory records.

The database stores persistent users, optional Google subject identities, magic-link login tokens, organizations, organization memberships, and organization invitations. User email is unique and canonical so Google and magic-link identities auto-merge when their emails match. Membership roles are `owner` and `member`; both can edit workspace data, while owners manage members and organization deletion. Invitations live in `organization_invitations` with hashed tokens, invited email, role, inviter, expiration, and accepted/canceled timestamps. Magic-link tokens live in `magic_link_tokens` with hashed single-use tokens, email, expiration, and used timestamps. Company attributes are flattened onto the `organizations` table, service/privacy/infrastructure/security/data handling/access sections use one-to-one profile tables, organization data categories live in `organization_data_types`, and business activities live in `business_activities` with service assignments in `service_business_activities` and processed data-type mappings in `business_activity_data_types`. Editable profile values are nullable so `null` represents unanswered fields and explicit empty arrays, empty strings, `false`, and zero remain intentional answers. Profile-owned lists that need this distinction use nullable JSON columns instead of Prisma scalar lists. Purpose and legal-basis values belong to business activities, not data types. Vendors are defined once per organization in `vendors`; service-specific processing details live in `service_vendor_uses`, and `vendor_data_types` links service-vendor uses to the organization data categories they process. Controlled vocabulary definitions loaded from Airtable live in `system_code_sets` and `system_codes`; non-system sets are cloned into `organization_code_sets` and `organization_codes` for organization editing, and removed organization codes are retained with `deleted_at` so system defaults are not backfilled after explicit deletion. Countries live in the separate `countries` table and use ISO alpha-2 codes. Development security, vulnerability management, and incident response are stored on `security_profiles`; access controls remain on `access_profiles`, and infrastructure security controls remain on `infrastructure_profiles`. Infrastructure system selections and privacy analytics/advertising/newsletter selections use `organization_providers.provider_id` and `organization_providers.system_type` instead of provider-name columns on profile tables. Editable organization templates live in `templates`; copied templates retain their source system template slug, scratch templates leave it null, and all templates store per-template policy metadata used by generated documents. Generated markdown documents live in `documents`, keep a source hash and required JSON source fingerprint for referenced-variable stale detection and stale reasons, and store private PDF export object paths for authenticated downloads. API-facing payloads remain nested and are validated at API boundaries with shared Zod schemas.

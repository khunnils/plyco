# Database Package Architecture

`@plyco/db` owns Prisma and database mapping.

The database stores persistent Google-backed users, organizations, and organization memberships. Membership roles are `owner` and `member`; both can edit workspace data in the current product surface. Company attributes are flattened onto the `organizations` table, service/privacy/infrastructure/data handling/access sections use one-to-one profile tables, organization data categories live in `organization_data_types`, and business activities live in `business_activities` with service assignments in `service_business_activities`. Editable profile values are nullable so `null` represents unanswered fields and explicit empty arrays, empty strings, `false`, and zero remain intentional answers. Profile-owned lists that need this distinction use nullable JSON columns instead of Prisma scalar lists. Purpose and legal-basis values belong to business activities, not data types. Vendors are defined once per organization in `vendors`; service-specific processing details live in `service_vendor_uses`, and `vendor_data_types` links service-vendor uses to the organization data categories they process. Controlled vocabulary definitions loaded from Airtable live in `system_code_sets` and `system_codes`; non-system sets are cloned into `organization_code_sets` and `organization_codes` for organization editing, and removed organization codes are retained with `deleted_at` so system defaults are not backfilled after explicit deletion. Countries live in the separate `countries` table and use ISO alpha-2 codes. Security-control detail is stored on the existing `access_profiles` and `infrastructure_profiles` rows rather than a separate security table. Infrastructure system selections and privacy analytics/advertising/newsletter selections use `organization_providers.provider_id` and `organization_providers.system_type` instead of provider-name columns on profile tables. Editable organization templates live in `templates`; copied templates retain their source system template slug, scratch templates leave it null, and all templates store per-template policy metadata used by generated documents. Generated markdown documents live in `documents`, keep a source hash for stale-state detection, and store private PDF export object paths for authenticated downloads. API-facing payloads remain nested and are validated at API boundaries with shared Zod schemas.
